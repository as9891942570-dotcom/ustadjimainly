Project Overview

ustadji is a home-services marketplace frontend for the Indian market. Customers browse skilled workers (plumbers, electricians, carpenters, etc.), submit service requests, and track bookings. Workers use a separate portal to register, complete Aadhaar-based KYC, and manage their profiles.

Key architectural fact This repository contains only the Next.js frontend. All persistence is handled by two separate REST backends hosted on Render. There are no Next.js API routes, no ORM, and no server-side database in this codebase.

System Architecture

Authentication

Two independent auth systems run in parallel — customer and worker — each with its own localStorage keys and axios interceptors.

sequenceDiagram
  participant U as User Browser
  participant C as AuthContext
  participant LS as localStorage
  participant API as Customer API (/backend)

  U->>C: login(email, password)
  C->>API: POST /auth/login
  API-->>C: { user_id, ... }
  C->>LS: setToken(user_id), setUser(user)
  Note over C,API: All requests: Authorization: Bearer {user_id}
  API-->>C: 401 Unauthorized
  C->>LS: clearAuthStorage()
  C->>U: redirect to /login

Customer Auth

  • Token stored as ustadji_token — value is the numeric user_id, not a JWT
  • Request interceptor attaches Authorization: Bearer {user_id}
  • 401 responses clear storage and redirect to /login
  • ProtectedRoute redirects unauthenticated users client-side

Worker Auth

  • Separate keys: ustadji_worker_token, ustadji_worker_user, ustadji_worker_profile_id
  • 401/403 → clear storage, redirect to /worker/login
  • WorkerProtectedRoute blocks customers (redirects to /dashboard)
Security concern Using the raw user ID as a bearer token is trivially forgeable if the backend only validates the ID without a signed JWT or session secret. All tokens live in localStorage, which is vulnerable to XSS. There are no httpOnly cookies, refresh tokens, or CSRF protection.

Booking Flow

Path A — Quick Request (works end-to-end)

Available on /services and /contact. Submits a simple service request to the backend:

POST /auth/request-service/{userId}
Body: { service_type, description, address, ... }

These requests appear on the customer /dashboard.

Path B — Full Booking Wizard (UI only — critical gap)

The 8-step /book-service flow collects rich booking data (service, preference, slot, address, payment method) but never calls the backend on confirmation.

Critical gap: OrderConfirmation does not persist bookings components/service/OrderConfirmation.tsx only shows a success modal and redirects home after 5 seconds. No api.post() is made. Confirmed wizard bookings never reach the server and will not appear on the dashboard.

Client-Side Pricing

utils/priceCalculation.ts hardcodes INR rates per service/preference. Prices are not fetched from the backend catalog and may drift from actual backend pricing.

Recommended Changes

Prioritized roadmap to move ustadji from a UI prototype toward a production-ready marketplace.

Phase 1 — Critical (make core flows work)

Priority: Critical

1. Wire booking wizard to the backend

Add a POST call in the payment/confirmation step (e.g. extend /auth/request-service/{userId} or add a new /bookings endpoint) with the full wizard payload: service, preference, booking type, slot, address, and amount. Update OrderConfirmation.tsx to submit before showing success. Ensure confirmed bookings appear on /dashboard.

Priority: Critical

2. Fix authentication model

Replace user-ID-as-token with proper JWT or session tokens issued by the backend. Move tokens to httpOnly cookies via a Next.js BFF layer (app/api/ routes) to eliminate XSS token theft. Add token refresh and CSRF protection.

Priority: Critical

3. Sync profile updates to the backend

app/profile/page.tsx currently calls updateUser() which only writes to localStorage. Add a PUT /auth/customers/{userId} (or equivalent) API call so profile edits persist across sessions and devices.

Phase 2 — Architecture (stabilize & unify)

Priority: Medium

4. Add a Backend-for-Frontend (BFF) layer

Create app/api/ route handlers that proxy to one or both backends. This centralizes auth, hides backend URLs, enables server-side session management, and provides a single API surface for the frontend. Long-term: consolidate the two Render backends into one service.

Priority: Medium

5. Add Next.js middleware for route protection

Create middleware.ts to check auth cookies server-side before rendering protected pages. Eliminates the flash-of-unauthenticated-content and enables SSR with user context.

Priority: Medium

6. Unify booking paths

Merge the "Quick Request" modal and the full booking wizard into a single coherent flow. Quick request could become a simplified step-1 shortcut that feeds into the same backend endpoint and dashboard view.

Priority: Medium

7. Server-driven pricing and addresses

Fetch service rates from the backend catalog instead of hardcoding in priceCalculation.ts. Persist customer addresses via the profile API instead of localStorage only.

Phase 3 — Product & Quality

Priority: Medium

8. Add worker marketplace listing

Implement customer-facing worker search/browse (the getAllWorkers() stub currently returns empty). Connect approved workers to incoming service requests for matching and assignment.

Priority: Medium

9. Adopt React Query or SWR

Replace ad-hoc useServices() calls with a data-fetching library for caching, deduplication, background refresh, and consistent loading/error states across home, services, and booking pages.

Priority: Low

10. Developer experience improvements

Replace boilerplate README with project docs. Add .env.example. Add ESLint CI, unit tests for utilities, and E2E tests for auth and booking flows. Fix duplicate "use client" directive in OrderConfirmation.tsx.
